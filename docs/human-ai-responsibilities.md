# 人間とAIの役割分担

## 基本原則

このシステムは**人間とAIの協働**を前提としています。完全な自動化を目指すのではなく、それぞれの強みを活かした役割分担を明確にします。

### 人間の強み
- ドメイン知識
- ビジネス判断
- 文脈理解
- 優先順位の判断
- リスク受容の意思決定
- ステークホルダー調整

### AIの強み
- パターン認識
- 網羅的な検証
- 一貫性のチェック
- 過去事例との照合
- 大量データの処理
- 24時間稼働

## 詳細な役割分担

### フェーズ1: 入力準備

#### 人間の役割

##### 1. 要件文書の準備 ★必須★
```yaml
responsibility: 人間
tasks:
  - 要件定義書、仕様書、チケットなどの文書を用意
  - 分析対象の範囲を決定
  - 文書の品質が最低限のレベルに達しているか確認
  
quality_criteria:
  - 文章として読めるレベル
  - 主語述語が存在する
  - 何について書かれているかが分かる
  
not_required:
  - 完璧な日本語である必要はない
  - 技術用語の統一は不要
  - フォーマットの整形は不要
```

##### 2. コンテキスト情報の提供 ★推奨★
```yaml
responsibility: 人間
tasks:
  - プロジェクトの背景説明
  - ドメイン情報（EC、金融、医療など）
  - 制約条件（予算、期限、品質優先度）
  - チーム構成
  
why_important:
  AIはコンテキストがあることで、より適切な分析と提案ができる
  
examples:
  good: "ECサイトのカート機能。予算タイト、品質重視、チーム5名"
  acceptable: "ECサイトのカート機能"
  poor: "システムの開発"（何のシステムか不明）
```

##### 3. 分析オプションの選択 ★任意★
```yaml
responsibility: 人間
tasks:
  - 抽出するカテゴリの選択（デフォルト: 全て）
  - 信頼度の閾値設定（デフォルト: 0.5）
  - 出力形式の指定（デフォルト: JSON）
  
when_to_customize:
  - 特定のカテゴリだけ確認したい（例: セキュリティのみ）
  - 時間がないので高信頼度のみに絞りたい
  - レポート形式を指定したい
```

#### AIの役割

##### 1. 入力の検証と正規化 ★自動★
```yaml
responsibility: AI
tasks:
  - 文字コードの検出と変換
  - 言語の自動検出
  - フォーマットの識別（プレーンテキスト、Markdown、JSONなど）
  - ノイズの除去（不要な空白、制御文字など）
  
human_intervention_needed:
  - 文字化けが激しくて解読不可能
  - 複数言語が混在していて主言語が特定できない
  - 画像やバイナリデータが含まれている
```

### フェーズ2: 要件解析（Module 1）

#### 人間の役割

##### 1. 解析結果のレビュー ★推奨★
```yaml
responsibility: 人間
tasks:
  - エンティティの抽出結果が妥当かチェック
  - 誤認識の修正（例: 「リンゴ」を商品ではなく会社名と認識した）
  - 重要度の調整
  
when_to_review:
  - 専門用語が多いドメイン
  - 文脈依存の解釈が必要
  - AIの信頼度が低い項目
  
skip_ok_if:
  - 一般的なドメイン（EC、CRMなど）
  - 文書が構造化されている
  - 次のステップで修正可能
```

#### AIの役割

##### 1. 構造化と抽出 ★自動★
```yaml
responsibility: AI
tasks:
  - 文章を文単位に分割
  - エンティティ（名詞）の抽出
  - アクション（動詞）の抽出
  - 関係性の推論
  - 制約条件の識別
  
automation_level: 完全自動
  
human_review_trigger:
  - confidence < 0.5 の項目が30%以上
  - 矛盾が検出された場合
  - ドメイン知識が不足していると判断した場合
```

### フェーズ3: 未定義要素の抽出（Module 2）

#### 人間の役割

##### 1. 未定義要素の妥当性確認 ★重要★
```yaml
responsibility: 人間
tasks:
  - 指摘された未定義要素が本当に未定義か確認
  - 「暗黙的に定義されている」項目の除外
  - 「この文書のスコープ外」項目の除外
  
why_important:
  AIは文書に書かれていないことを検出するが、
  それが「暗黙の了解」なのか「本当に未定義」なのかは人間が判断すべき
  
examples:
  ai_detection: "ユーザーIDの型が定義されていない"
  human_judgment_1: "既存システムと同じUUIDを使う（暗黙の了解）" → 除外
  human_judgment_2: "確かに未定義。明示的に文書化すべき" → 採用
```

##### 2. 質問の追加・修正 ★推奨★
```yaml
responsibility: 人間
tasks:
  - AIが生成した質問に追加
  - 質問の表現を修正（より具体的に、ステークホルダーに伝わる形に）
  - 優先順位の調整
  
examples:
  ai_generated: "在庫確認のタイミングは？"
  human_refined: "在庫確認は、①カート追加時 ②購入確定時 ③どちらのタイミングで行うか？ また、確認から購入確定までの間に他ユーザーが購入した場合の挙動は？"
```

#### AIの役割

##### 1. パターンベース検出 ★自動★
```yaml
responsibility: AI
tasks:
  - 分類テンプレートに基づく検出
  - 曖昧表現の検出（「速い」「多い」「適切に」など）
  - 欠落パターンの検出（条件文で例外処理の記述なし、など）
  - 矛盾の検出
  
automation_level: 完全自動
  
output_quality:
  - 誤検知（False Positive）: 許容 → 人間がフィルタ
  - 検出漏れ（False Negative）: 最小化 → 網羅性重視
```

##### 2. 質問の自動生成 ★自動★
```yaml
responsibility: AI
tasks:
  - 未定義要素ごとに解消のための質問を生成
  - 確認すべき観点を列挙
  - 過去の類似ケースを参照
  
customization_by_human:
  - 質問の表現を修正
  - 質問の追加・削除
  - 優先順位の変更
```

### フェーズ4: リスク分析（Module 3）

#### 人間の役割

##### 1. リスク評価の調整 ★重要★
```yaml
responsibility: 人間
tasks:
  - AIのリスク評価が適切かレビュー
  - プロジェクト固有の事情を反映
  - ビジネス的な影響度の評価
  
why_important:
  AIは一般的なリスクパターンで評価するが、
  プロジェクト固有の事情（例: このプロジェクトでは品質より速度優先）は人間が判断
  
examples:
  ai_assessment: "在庫マイナスのリスク: 高"
  human_adjustment_1: "初期は少数ユーザーなので中リスクに下げる"
  human_adjustment_2: "当社は在庫管理で過去に炎上したので最高リスクに上げる"
```

##### 2. リスク受容の判断 ★人間のみ★
```yaml
responsibility: 人間（AIは提案のみ）
tasks:
  - リスクを受容するか、解消するか決定
  - 受容する場合の条件と監視方法を決定
  - 意思決定の記録
  
decision_criteria:
  - コスト vs ベネフィット
  - ビジネス優先度
  - 競合状況
  - 規制要件
  
ai_role:
  - 判断材料の提供（過去事例、コスト試算）
  - 代替案の提示
  - 決定はしない
```

#### AIの役割

##### 1. リスクシナリオの生成 ★自動★
```yaml
responsibility: AI
tasks:
  - 未定義要素が引き起こす具体的な問題シナリオを生成
  - 過去の失敗事例との照合
  - 影響範囲の特定
  - 発生確率の推定
  
automation_level: 完全自動（確率的推論）
  
accuracy:
  - 網羅性: 高（考えられるシナリオを広く提示）
  - 精度: 中（プロジェクト固有事情は反映できない）
```

##### 2. 4軸評価の算出 ★自動★
```yaml
responsibility: AI
tasks:
  - 発生確率の推定
  - 影響度の評価
  - 発見困難度の評価
  - 修正コストの見積もり
  
basis:
  - 過去の統計データ
  - 一般的なプロジェクトの傾向
  - ドメイン固有のパターン
  
limitations:
  - チームのスキルレベルは考慮できない
  - 政治的要因は評価できない
  - 個別プロジェクトの特殊事情は反映されない
```

### フェーズ5: 修正指示の生成（Module 4）

#### 人間の役割

##### 1. アクションの決定 ★人間のみ★
```yaml
responsibility: 人間（AIは提案のみ）
tasks:
  - 推奨アクションを実行するか決定
  - 担当者のアサイン
  - スケジュールへの組み込み
  - ステークホルダーとの調整
  
decision_factors:
  - リソースの可用性
  - 他タスクとの優先順位
  - 政治的配慮
  - ビジネス判断
  
ai_cannot_do:
  - 人の調整
  - スケジュールの変更
  - 予算の承認
  - ステークホルダーへの説明
```

##### 2. 代替案の選択 ★人間のみ★
```yaml
responsibility: 人間
tasks:
  - AIが提示した複数の代替案から選択
  - リスク受容の場合の監視計画策定
  - 延期の場合のトリガー条件設定
  
examples:
  option_1: "今すぐ設計に戻って解消（コスト高、リスク低）"
  option_2: "MVP版では受容、v2で対応（コスト低、リスク中）"
  option_3: "発生時の対応手順を用意（コスト低、リスク高）"
  
  human_choice: プロジェクトのフェーズと制約に応じて選択
```

#### AIの役割

##### 1. 工程判定と推奨アクション生成 ★自動★
```yaml
responsibility: AI
tasks:
  - どの工程で解消すべきか判定
  - 具体的なアクションステップの生成
  - 必要な工数の見積もり
  - 代替案の提示
  
automation_level: 完全自動
  
output:
  - メインの推奨アクション
  - 2-3個の代替案
  - それぞれのPros/Cons
  - 工数とリスクのトレードオフ
```

##### 2. クイックウィンの特定 ★自動★
```yaml
responsibility: AI
tasks:
  - 短時間で効果が大きい項目を特定
  - ROIの高い順にソート
  - 即座に対応可能な項目を強調
  
value:
  時間がない場合でも、最小の工数で最大の効果を得られる項目を提示
```

### フェーズ6: 実行とトラッキング

#### 人間の役割

##### 1. 実際の解消作業 ★人間のみ★
```yaml
responsibility: 人間
tasks:
  - ステークホルダーへのヒアリング
  - 要件の明確化
  - 設計の作成
  - ドキュメントの更新
  - コードの実装
  
ai_cannot_do:
  - ビジネス判断
  - ドメインエキスパートとの対話
  - 創造的な設計
  - 実装
```

##### 2. 決定の記録 ★重要★
```yaml
responsibility: 人間
tasks:
  - 未定義要素をどう解消したか記録
  - リスクを受容した場合の理由を記録
  - 決定者と日時を記録
  
why_important:
  - 後で「なぜこうしたか」を思い出せる
  - 担当者が変わっても経緯が分かる
  - 同じ議論を繰り返さない
  - 監査証跡になる
  
ai_support:
  - テンプレートの提供
  - 記録忘れのリマインド
  - 記録内容の構造化
```

#### AIの役割

##### 1. 進捗のトラッキング ★自動★
```yaml
responsibility: AI
tasks:
  - 未解消の項目をダッシュボード表示
  - 期日が近い項目の通知
  - 完了率の可視化
  
automation_level: 完全自動（人間が記録を入力すれば）
```

##### 2. 再分析のトリガー ★自動★
```yaml
responsibility: AI
tasks:
  - 文書が更新されたら再分析を提案
  - 新たな未定義要素の検出
  - 以前の指摘が解消されたか確認
  
when:
  - 文書の更新検知
  - 工程が進んだとき
  - 定期的（週次など）
```

## 判断マトリクス

### 誰が何を判断するか

| 判断項目 | 人間 | AI | 根拠 |
|---------|------|-----|------|
| **要件の正しさ** | ◎ | △ | ビジネス判断が必要 |
| **未定義要素の検出** | △ | ◎ | 網羅的なチェックが必要 |
| **リスクの存在** | △ | ◎ | パターン認識が有効 |
| **リスクの評価** | ◎ | △ | プロジェクト固有事情が影響 |
| **リスクの受容** | ◎ | × | 経営判断 |
| **工程の判定** | △ | ◎ | ルールベースで判定可能 |
| **担当者のアサイン** | ◎ | × | 組織の事情が関係 |
| **スケジュール調整** | ◎ | × | 複雑な制約条件がある |
| **ステークホルダー調整** | ◎ | × | 対人コミュニケーションが必要 |
| **進捗の可視化** | × | ◎ | 自動化すべき |

◎: 主担当、△: 補助、×: 実施不可

## 情報の流れ

### 人間 → AI

```yaml
input_from_human:
  必須:
    - 要件文書
  
  推奨:
    - プロジェクトコンテキスト
    - ドメイン情報
    - 制約条件
  
  任意:
    - 過去の失敗事例
    - ドメイン固有のルール
    - カスタム分類テンプレート
  
  フィードバック:
    - 誤検知の指摘
    - 検出漏れの報告
    - リスク評価の修正
```

### AI → 人間

```yaml
output_to_human:
  構造化データ:
    - 未定義要素リスト
    - リスク評価
    - 推奨アクション
  
  可視化:
    - ダッシュボード
    - グラフ
    - マトリクス
  
  自然言語:
    - サマリーレポート
    - 質問リスト
    - 推奨理由の説明
  
  アラート:
    - 重大なリスクの通知
    - 期日のリマインド
    - 矛盾の検出
```

## 協働のベストプラクティス

### 1. AIを"アシスタント"として扱う

```yaml
good_approach:
  - AIの出力をレビューする
  - 疑問があれば人間が判断する
  - AIの強みを活かして人間の負荷を減らす

bad_approach:
  - AIの出力を無批判に受け入れる
  - AIに全ての判断を委ねる
  - AIの指摘を全て無視する
```

### 2. フィードバックループを回す

```yaml
cycle:
  1. AIが分析
  2. 人間がレビュー
  3. 人間が修正・追加
  4. AIが学習（フィードバック）
  5. 次回の精度向上

key_point:
  人間のフィードバックをAIに返すことで、
  プロジェクト固有の知識がシステムに蓄積される
```

### 3. 役割を明確に理解する

```yaml
human_responsibility:
  - 意思決定
  - ビジネス判断
  - ステークホルダー調整
  - 創造的な問題解決

ai_responsibility:
  - 情報収集
  - パターン認識
  - 候補の提示
  - 進捗の可視化

both:
  - 品質の向上（人間が設計、AIが検証）
  - リスクの発見（AIが検出、人間が評価）
```

## スキルレベル別の使い方

### 初心者（システムに不慣れ）

```yaml
recommendation:
  - デフォルト設定で実行
  - AIの出力を全て確認
  - 分からない項目はスキップせず質問
  
ai_support:
  - 用語の説明
  - 推奨設定の提示
  - ステップバイステップのガイド
```

### 中級者（プロジェクト経験あり）

```yaml
recommendation:
  - カテゴリを絞って分析
  - リスク評価を調整
  - クイックウィンに集中
  
ai_support:
  - カスタマイズ可能な設定
  - 詳細なリスク分析
  - 複数の代替案提示
```

### 上級者（アーキテクト、PM）

```yaml
recommendation:
  - カスタム分類テンプレート作成
  - ドメイン知識の追加
  - 過去の分析結果の活用
  
ai_support:
  - API経由での自動化
  - CI/CD統合
  - 組織全体への知識共有
```

## まとめ

### 成功の鍵

1. **AIは完璧ではない**ことを理解する
2. **人間の判断**が最終的に重要
3. **フィードバック**でシステムを育てる
4. **役割分担**を明確にする
5. **継続的に利用**することで精度が向上

### 避けるべき誤解

❌ AIが全てを解決してくれる  
✅ AIは人間を支援するツール

❌ AIの指摘は全て正しい  
✅ AIの指摘は確認が必要

❌ 一度使えば終わり  
✅ 継続的に使うことで価値が高まる
